# Альтернативные способы доступа к VPS без SSH

## Проблема
SSH недоступен (порт 22 закрыт или sshd не запущен), консоль VPS недоступна.

## Решения (пробуем по порядку)

### 1. Через GitHub Actions (другой IP)
**Actions** → **Fix via File Upload** → **Run workflow**

Workflow попробует:
- Прямой SSH (может сработать с IP GitHub)
- Telnet на порт 22
- HTTP PUT (если nginx разрешает)

### 2. Через провайдера VPS API
Если у вас Hetzner/DigitalOcean/AWS/etc — зайдите в панель провайдера и:
- Найдите API ключи
- Используйте их для выполнения команд через API
- Или найдите опцию "Execute Command" / "Run Script" в панели

### 3. Через веб-интерфейс провайдера
- Панель управления VPS → **"Console"** / **"VNC"** / **"Web Terminal"**
- Если консоль не открывается — попробуйте другой браузер или режим инкогнито
- Или используйте мобильное приложение провайдера

### 4. Через существующие процессы (если боты работают)
Если Telegram-боты работают, они могут иметь доступ к системе. Попробуйте:
- Отправить боту команду `/admin` или `/exec` (если такая функция есть)
- Проверить логи ботов — может быть там есть способ выполнить команду

### 5. Через файловую систему (если есть доступ к файлам)
Если есть способ загрузить файл на сервер (через веб-интерфейс провайдера или другой способ):
1. Создайте файл `/tmp/fix-ssh.sh` с содержимым:
```bash
#!/bin/bash
systemctl start sshd
systemctl enable sshd
systemctl start nginx
cd /var/www/like-that 2>/dev/null && (pm2 restart like-that 2>/dev/null || pm2 start npm --name like-that -- start)
pm2 save
```
2. Выполните через cron или другой механизм

### 6. Через провайдера поддержку
Свяжитесь с поддержкой провайдера VPS и попросите:
- Запустить SSH демон
- Или выполнить команды от вашего имени

### 7. Через резервное копирование/восстановление
Если есть резервная копия сервера — восстановите её, но это может привести к потере данных.
